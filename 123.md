# CalculatorX 项目总结

> 本文档由项目源码全面审查后生成，最后更新于 2026-06-12。

---

## 一、项目概述

**CalculatorX**（CalcX 计算器）是一款面向 **HarmonyOS** 平台的专业符号计算器应用，支持**精确符号运算**与**高精度数值计算**双模式输出。项目采用 **"前端 UI（ArkTS）+ Web 渲染（MathLive）+ 双 C++ 引擎（Giac / SymEngine）"** 三层架构，覆盖从基础四则运算到高等数学（微积分、极限、代数化简）的核心功能。

- **包名**：`com.startyi.calcx`
- **版本**：1.0.0 (versionCode: 1000000, Beta 0.1.2)
- **许可证**：GNU GPL v3.0
- **作者**：易睿 (Yi Rui)
- **官网**：https://calcx.startyi.com
- **SDK**：HarmonyOS 6.1.1(24)，兼容 6.1.0(23)
- **开发环境**：DevEco Studio
- **支持设备**：Phone / Tablet

## 二、核心架构

### 三层架构体系

```
┌─────────────────────────────────────────────────────┐
│  ① UI 控制层 (ArkTS / ArkUI)                         │
│  · 声明式 UI 原生键盘                                  │
│  · ⇧Shift 状态机拦截、角度制/弧度制管理               │
│  · 统一调度下层计算资源                                │
├─────────────────────────────────────────────────────┤
│  ② 渲染引擎层 (Web Component + MathLive)              │
│  · 出版级 LaTeX 数学公式高清渲染                       │
│  · MathJSON 数据降维：二维 LaTeX → 结构化 AST          │
│  · 通过 runJavaScript 跨端 DOM 操作与渲染指令通信      │
├─────────────────────────────────────────────────────┤
│  ③ 双引擎计算层 (C++ & N-API)                        │
│  · Giac 引擎：符号计算（微积分/极限/方程求解/化简）    │
│  · SymEngine 引擎：高精度数值计算与 AST 解析           │
│  · FastMath 模块：O(1) 超大数计算（对数+斯特林近似）   │
│  · ErrorHandler 模块：精准异常拦截与前端友好报错       │
└─────────────────────────────────────────────────────┘
```

### 计算流程

```
用户按键 → ArkTS handleButtonClick()
  → WebView.runJavaScript() 注入 LaTeX 指令
  → MathLive 渲染为 MathJSON AST
  → N-API 传递 JSON 到 C++ engine.cpp
  → parseAST() 递归解析为 SymEngine 表达式树
  → 精度控制/格式化（根据 precision 模式）
  → LaTeX 字符串返回前端渲染
```

## 四、功能矩阵

### 4.1 基础运算
| 功能 | 实现方式 |
|------|----------|
| 四则运算（+ − × ÷） | SymEngine 表达式树 |
| 分数（\frac） | MathLive → MathJSON → SymEngine Rational |
| 百分号（%） | `parseAST` → ÷100 |
| 取模（mod） | `a - b * floor(a/b)` |
| 幂运算（x², x³, xⁿ, 10ˣ, eˣ） | SymEngine::pow |
| 开方（√, ³√, ⁿ√） | `pow(x, 1/n)` |
| 绝对值 | SymEngine::abs |
| 倒数（x⁻¹） | 幂运算特例 |

### 4.2 三角函数
| 功能 | 特点 |
|------|------|
| sin / cos / tan | 支持角度制/弧度制动态切换 |
| arcsin / arccos / arctan | 反向输出也受角度模式影响 |
| 度分秒（DMS）输入 | 智能状态机：° → ' → " 循环 |
| 度分秒格式化输出 | 自动渲染为 `12°34'56"` |

### 4.3 高等数学
| 功能 | 引擎 | 说明 |
|------|------|------|
| 求导（d/dx） | SymEngine | `body.diff(symbol("x"))` |
| 不定积分 | **Giac**（主） | `integrate(expr, var)` → LaTeX |
| 定积分 | **Giac**（符号）+ **Simpson**（数值兜底） | Giac 失败自动降级到辛普森1/3 |
| 极限 | **Giac** | `limit(expr, var, target)` |
| 求和（Σ） | SymEngine | 循环代入，上限≤10000 |
| 求积（Π） | SymEngine | 同上，上限≤10000 |

### 4.4 组合数学
| 功能 | 特点 |
|------|------|
| 阶乘（!） | ≤5000 用Gamma函数，>5000 用斯特林近似 |
| 组合数（nCr） | 5种显示风格可选（国际学术/C角标/左上标/左下标/平铺） |
| 排列数（nPr） | 5种显示风格 + A/P 双字母体系 |
| 超大数处理 | FastMath 模块，支持 10^9000000000000000000 级别 |

### 4.5 输出格式
| 模式 | precision 码 | 说明 |
|------|-------------|------|
| 自动模式 | -1 | 含小数→小数；纯整数→保留精确形式 |
| 小数模式（精度可调） | 0~16 | 固定精度输出 |
| 高精度小数 | -2 | 12位有效数字，去除尾部零 |
| 假分数 | -3 | 强制精确分数表示 |
| 带分数 | -4 | 整数部分 + 真分数，如 `2\frac{1}{3}` |
| 度分秒 | -5 | 强制 DMS 格式化 |
| 科学计数法（自动） | — | 结果 ≥1e15 或 ≤ 1e-5 时触发 |

### 4.6 交互体验
- **Shift** 二级功能键盘（30个按键支持 Shift 切换）
- **触感反馈**：4种震感曲线（自动差异化/清脆/轻柔/厚重）
- **长按菜单**：S⇄D 按钮长按弹出格式选择（小数/分数/带分数/度分秒）
- **长按连发**：退格键和光标键支持长按连续操作（500ms 延迟 + 80ms 间隔）
- **深/浅色模式**：跟随系统自动切换，支持手动覆盖
- **S⇄D 一键切换**：精确形式 → 小数形式互转
- **Ans**：自动引用上一次计算结果

## 五、技术亮点

### 5.1 数据降维设计
MathLive 生成的 **MathJSON** 是结构化 AST，将"二维 LaTeX 公式" 变为"一维 JSON 树"，大幅降低 C++ 引擎的解析复杂度。这是一个精妙的工程权衡：用 Web 引擎做复杂解析，用 C++ 做高性能计算。

### 5.2 双引擎互补策略
- **Giac**：处理 SymEngine 不擅长的极限、积分等高等数学推导，并通过 `MAGICGIACRESULT` 标记回传渲染。
- **SymEngine**：处理日常运算、AST 解析、精确分数运算。
- **Simpson 数值积分**：当 Giac 符号积分失败时自动降级为数值近似。

### 5.3 超大数计算（FastMath）
- 基于斯特林近似公式 `log10(n!) ≈ 0.5·log10(2πn) + n·log10(n/e)` 在 O(1) 时间内计算阶乘的量级。
- 用 `MAGICBASETEN` 幽灵变量在 SymEngine 表达式树中表示 `A × 10^B`，绕过 64 位限制。

### 5.4 自定义异常机制
`ErrorHandler.h` 定义了 6 种业务错误码（除零、定义域、溢出、语法、超时、DMS格式），统一映射为前端可识别的 `Error:XXX` 字符串，保障前端无闪退。

### 5.5 跨平台 C++ 构建
通过 CMake + `FetchContent` 自动拉取 SymEngine（从 GitHub），配合本地离线 tar.gz 包（Boost、Giac），并手写 `glibc_stub.c` 补齐 musl libc 与 glibc 的符号差异（`__fprintf_chk`、`__ctype_b_loc`），实现了在 HarmonyOS 上编译 Linux 生态的工业级 CAS 库。

### 5.6 优雅的 UI 架构
- 键盘拆分为 `TopKeyboard`（科学计算）+ `BottomKeyboard`（基础运算），职责清晰
- `CalculatorConfigs.ets` 集中管理键盘布局、样式映射、SVG 资源、首选项键名
- 组合/排列样式通过底部半模态面板 `MathStyleSheet` 实时预览切换

## 八、可改进之处

### 🔴 高优先级

#### 1. 安全性：签名密钥硬编码
[build-profile.json5](build-profile.json5) 中包含完整的 `.p12` 密钥库密码、密钥别名密码、证书路径等敏感信息，以明文形式提交到 Git。

**建议**：
- 将签名配置移至 `local.properties` 或环境变量
- 将 `build-profile.json5` 中的敏感字段改为变量引用
- 检查 git 历史，确保密钥未被推送到公开仓库

#### 2. 安全隐患：构建路径硬编码
多处使用 `E:/Program Files/HUAWEI/Keys/` 和 `E:/Program Files/HUAWEI/DevEco Studio/sdk` 等绝对路径，无法在其他开发环境构建。

**建议**：使用相对路径或环境变量，在 README 中说明开发者环境配置步骤。

#### 3. 缺少单元测试
虽然已引入 `@ohos/hypium` 和 `@ohos/hamock` 测试依赖，但项目中未发现任何测试文件（`src/test/` 或 `src/ohosTest/` 目录似乎不存在或为空）。

**建议**：
- 为 C++ 引擎的 `parseAST` 函数编写单元测试（覆盖各种 MathJSON 节点类型）
- 为 `FastMath` 模块编写精度验证测试
- 参考 [docs/计算的审查.md](docs/计算的审查.md) 中的测试清单，建立系统化测试套件

#### 4. calculator.html 存在重复函数定义
[calculator.html:127](entry/src/main/resources/rawfile/calculator.html#L127) 和 [calculator.html:185](entry/src/main/resources/rawfile/calculator.html#L185) 两处定义了 `window.insertDMS`，第二个定义会覆盖第一个，且两个版本的逻辑有细微差异（新版增加了"秒后拦截数字"逻辑，但旧版的一些边界情况处理被移除）。

**建议**：合并为一个函数，保留两个版本的最佳逻辑。

### 🟡 中优先级

#### 5. engine.cpp 过于庞大
[engine.cpp](entry/src/main/cpp/engine.cpp) 单文件 842 行，包含了 JSON 解析、AST 转换、Giac 调用、数值格式化、N-API 入口等所有后端逻辑。

**建议**：拆分为以下模块：
- `parser.cpp` / `parser.h`：`parseAST()` 递归下降解析器
- `formatter.cpp` / `formatter.h`：`formatLargeIntegerToScientific()` 等输出格式化函数
- `giac_bridge.cpp` / `giac_bridge.h`：`evaluateWithGiac()` 和 `buildGiacCommand()`
- `engine.cpp`：仅保留 `Calculate()` N-API 入口和调度逻辑

#### 6. 魔法字符串过多
代码中大量使用硬编码的魔法字符串，如：
- `"MAGICGIACRESULT"` / `"MAGICBASETEN"` — 用作 SymEngine 符号标记
- `"Error:DivByZero"` / `"Error:Domain"` 等 — 错误消息
- `"Unknown\\_"` / `"Invalid\\_Node"` — 未知节点标记

**建议**：定义命名常量（`constexpr` 或 `#define`），集中管理，避免拼写错误导致的难以排查的 bug。

#### 7. 缺少国际化（i18n）框架
所有用户可见字符串硬编码为中文（设置页）或英文（错误消息）。虽然当前目标用户是中文用户，但随着项目发展可能需要支持多语言。

**建议**：将字符串提取到 `string.json` 资源文件中，使用 `$r('app.string.xxx')` 引用。

#### 8. Index.ets 中 LaTeX 正则替换逻辑繁杂
[Index.ets:203-248](entry/src/main/ets/pages/Index.ets#L203-L248) 包含大量正则表达式用于 LaTeX 预处理（组合/排列/求导/DMS），这些逻辑挤在主页面中不易维护。

**建议**：抽取为独立的 `LatexPreprocessor.ets` 工具模块，编写相应的单元测试。

### 🟢 低优先级

#### 9. 缺少 CI/CD 配置
没有 GitHub Actions、GitLab CI 或其他 CI/CD 配置文件。

**建议**：添加自动化构建验证、代码风格检查（clang-tidy 已配置但未集成到 CI）、自动化测试。

#### 10. 无障碍（Accessibility）支持缺失
键盘按钮和公式显示区未添加无障碍标签（`accessibilityText`、`accessibilityDescription` 等），视障用户无法使用。

**建议**：为按钮添加语义化无障碍描述，为公式渲染区提供文本替代。

#### 11. 未充分利用 TypeScript 严格类型
ArkTS 代码中使用了较多 `string | Resource` 联合类型和 `as` 类型断言，部分函数的参数和返回值可以定义更精确的接口类型。

**建议**：
- 为 MathJSON AST 节点定义 TypeScript 接口
- 为键盘配置项定义明确的数据类型
- 减少 `any` / `as` 的使用

#### 12. 计算超时与并发控制
README 提到"单次表达式计算设置超时阈值（如 5 秒）"，但 C++ 引擎中只通过迭代上限（如 `end - start > 10000`）做了硬性限制，未实现真正的 wall-clock 超时机制。连续快速点击 "=" 也缺少防抖/任务取消逻辑。

**建议**：
- 在 C++ 层引入异步超时检测
- 在前端增加防抖（debounce），避免计算任务堆积

#### 13. 缺少 CHANGELOG 与版本管理
项目无 `CHANGELOG.md`，无法追踪各版本的变更内容。

**建议**：遵循 [Keep a Changelog](https://keepachangelog.com/) 规范，在每次发布时记录变更。

#### 14. 日志系统可增强
当前使用 `console.info` / `console.warn` / `hilog.info` 等分散的日志方式，缺少统一的日志级别控制和格式化。

**建议**：封装统一的 Logger 工具类，支持日志级别过滤和格式化输出。

#### 15. 编译产物与缓存文件管理
`.hvigor/` 目录下包含大量构建日志（`build.log` 系列）、性能记录等运行时产物。这些文件在 `.gitignore` 中已忽略，但本地磁盘占用较大。

**建议**：定期清理构建缓存，或在 DevEco Studio 中执行 `Build → Clean Project`。
