## STL核心容器

STL（Standard Template Library，标准模板库）是 C++ 的“终极兵器库”。在实际开发中，除非有极其特殊的底层性能要求，否则我们绝对不会自己手写数组或链表，而是直接调 STL。

你的笔记里总结了四种最核心的容器。期末考试对这块的考察非常固定，通常集中在**底层数据结构**、**时间复杂度**，以及最容易写出 Bug 的**迭代器失效**问题上。

我们用大白话和生活场景，把这四个容器彻底扒个精光。

### 1. `vector`（动态数组）：不断搬家的“大仓库”

`vector` 是日常写代码绝对的绝对的主角（90% 的场景都在用它）。

- **底层逻辑：** 它是一块**连续**的内存。就像你租了一个仓库，所有的货物必须挨着放。
    
- **核心机制：扩容（倍乘）**
    
    这是考试**最高频的必考点**。
    
    假设你租了一个只能放 4 个箱子的仓库（`capacity = 4`），现在已经放满了（`size = 4`）。这时你又要塞进第 5 个箱子，会发生什么？
    
    `vector` 的做法极其暴力：
    
    1. 它去别的地方租一个**更大的新仓库**（通常是原来的 1.5 倍或 2 倍大，也就是容量变成 8）。
        
    2. 把老仓库的 4 个箱子**全搬过去**（拷贝/移动构造）。
        
    3. 把第 5 个箱子放进去。
        
    4. **把老仓库退租**（释放旧内存）。
        
- **致命考点：迭代器失效（迷路问题）**
    
    假设在搬家前，你给了朋友一张纸条（迭代器），上面写着：“我的第 3 个箱子放在老仓库的角落里”。
    
    结果 `vector` 刚刚经历了一次扩容（搬到了新仓库，老仓库被拆了）。你朋友拿着那张纸条去找箱子，结果发现那里变成了一片废墟——**这就叫迭代器失效**！
    
    **结论：** 只要 `vector` 发生了扩容，之前所有指向它的迭代器、指针、引用，**瞬间全部作废**。
    
- **避坑细节：`size()` vs `capacity()`**
    
    - `size()`：当前实际装了多少货物。
        
    - `capacity()`：这个仓库总共能装多少货物。
        
    - `reserve(n)`：提前租一个能装 n 个货物的空仓库（只改 capacity，不放实物，非常推荐在知道大概数量时使用，可以避免频繁搬家）。
        
    - `resize(n)`：不仅把仓库变大，还强行塞进去 n 个默认的空气盒子（同时改变 capacity 和 size）。
        

### 2. `list`（双向链表）：地下党的“单线联系”

- **底层逻辑：** 它在内存中是**极度分散**的。每一个节点就是一个小黑屋，里面除了放数据，还放了两张纸条：一张写着上一个节点在哪，一张写着下一个节点在哪。
    
- **绝对优势：插入和删除极快。** 如果在队伍中间插入一个人，`vector` 需要把后面所有人都往后挪一步（非常慢）。而 `list` 只需要改一下前后两个人的“纸条”上的地址就行了。
    
- **致命缺点：不能随机访问！**
    
    这是考卷上最爱出的改错题。你**绝对不能写** `myList[5]`。
    
    因为 `list` 是分散的，如果你想找第 5 个节点，你必须从第 1 个节点开始，挨个问纸条上的地址，走到第 5 个。所以它的查找时间复杂度是 $O(n)$。
    
- **关于迭代器失效：**
    
    因为 `list` 从来不“搬家”，所以在 `list` 里插入或删除元素时，**只有被删掉的那个节点的迭代器会失效**，其他所有人的迭代器都不受影响。
    

### 3. `map` vs `unordered_map`：字典与储物柜的对决

这两个都是**关联容器**，也就是里面存的都是键值对（`Key - Value`）。比如存学生的学号和名字：`{ "2024001" : "张三" }`。

期末考试一定会让你对比这两个容器的底层实现和适用场景。

- **`map`（传统字典）**
    
    - **底层原理：** 红黑树（一种极其平衡的二叉搜索树）。
        
    - **核心特征：自带排序！** 它会根据你的 Key 自动进行升序排列。当你遍历 `map` 时，输出的结果永远是有序的。
        
    - **时间复杂度：** 插入、删除、查找都是 $O(\log n)$。翻红黑树就像翻字典，每一次都折半查找，非常稳定。
        
- **`unordered_map`（带编号的储物柜）**
    
    - **底层原理：** 哈希表（Hash Table）。
        
    - **核心特征：绝对无序！** 你塞进去的顺序和遍历输出的顺序毫无关系。它通过一个哈希函数，把你的 Key 直接算出一个数字，然后丢进对应的“储物柜”格子里。
        
    - **时间复杂度：** 平均时间复杂度是极其霸道的 $O(1)$！也就是说，不管里面存了 10 个数据还是 100 万个数据，它找东西的速度是一样快的（除非运气极差发生了严重的哈希冲突，最坏会退化成 $O(n)$）。
        

**【考场通关秘籍：该选谁？】**

如果题目问：“需要频繁查找某个用户的 ID 是否存在，且对顺序没有要求，应该用什么容器？”

**秒选 `unordered_map`**，因为 $O(1)$ 的查找速度无敌。

如果题目问：“需要记录每天的温度，并且最后按日期先后顺序列印出来？”

**必须选 `map`**，因为它底层是红黑树，自带排序属性。

这就是 STL 的四驾马车。只要脑子里有画面（vector的连续大仓库、list的地下党纸条、map的红黑树字典、unordered_map的哈希储物柜），选择题和概念题绝对能拿满分。

## 虚函数

虚函数可以说是 C++ 面向对象编程里最精髓、也是期末考试必考的“重中之重”。它是实现多态（Polymorphism）的核心武器。

一句话概括它的作用：**虚函数让编译器在“程序运行的时候”，才根据对象的真实身份去决定调用哪个函数，而不是在编译时就定死。**

为了让你直观理解，我们不贴长篇大论，直接用代码构建一个期末考试最常考的场景——“父类指针指向子类对象”。

### 第一步：如果没有虚函数会怎样？（静态绑定）

假设我们在开发一个绘图软件，有一个基础形状类 `Shape`，和一个继承它的圆形类 `Circle`。

```C++
class Shape {
public:
    // 注意：这里暂时没有写 virtual
    void draw() { 
        cout << "画一个基础形状" << endl; 
    }
};

class Circle : public Shape {
public:
    // 子类重写了父类的方法
    void draw() { 
        cout << "画一个圆" << endl; 
    }
};
```

现在，到了期末考试最经典的“挖坑”环节。如果我们在主函数里这样写：

```C++
Circle myCircle;           // 真实身份：一个圆对象
Shape* ptr = &myCircle;    // 表象：用一个父类指针，指着这个圆

ptr->draw();               // 灵魂拷问：这里会打印什么？
```

**结果是：打印“画一个基础形状”。**

为什么？因为没有虚函数时，C++ 编译器是个“死脑筋”。它在编译代码时，一看 `ptr` 的类型是 `Shape*`，就直接把 `draw()` 绑定到了 `Shape` 类的函数上。它根本不在乎这个指针底层实际指着的是什么。这叫做**静态绑定（早绑定）**。

### 第二步：加上 `virtual` 后的魔法（动态绑定）

这显然不符合我们的预期——既然指着的是个圆，就应该画个圆啊！这时候，`virtual` 关键字就该登场了。

我们只需要在父类的函数前面加一个词：

```C++
class Shape {
public:
    // 加上了 virtual 关键字！
    virtual void draw() { 
        cout << "画一个基础形状" << endl; 
    }
};
```

子类代码完全不用动。这时候再去执行同样的操作：

`ptr->draw();`

**结果变成了：打印“画一个圆”。**

加了 `virtual` 之后，编译器变聪明了。它看到 `draw()` 是个虚函数，就会推迟决断，等到程序真正**运行**的时候，顺着指针去扒开对象的底牌：“哦，虽然你是用 `Shape` 指针指着的，但你本质上是个 `Circle` 对象啊！那我就调用 `Circle` 的 `draw()`。”这叫做**动态绑定（晚绑定）**。

### 期末加分项：底层是怎么实现的？（虚函数表）

如果在简答题里遇到“虚函数的实现原理”，你只要答出下面这两个词，老师基本就会给你满分：

1. **虚函数表 (vtable)**：只要类里面哪怕有一个虚函数，编译器就会在幕后为这个类偷偷建一张“表”，表里面存着所有虚函数的内存地址。
    
2. **虚表指针 (vptr)**：当你创建这个类的对象时，对象内部会隐式地多出一个指针，这个指针就指向那张虚函数表。
    

当你调用 `ptr->draw()` 时，程序实际上是：通过对象的 `vptr` 找到对应的 `vtable`，然后在表里查到了子类 `Circle::draw()` 的地址，最后才执行。这就是动态绑定的本质。

关于虚函数，期末通常还有两个高频且容易踩坑的衍生考点：一个是**纯虚函数（及抽象类）**，另一个是**虚析构函数**（解决内存泄漏的关键）。接下来你想先盘哪一个？

## 文件输入输出流

文件输入输出流（File I/O）在 C++ 中不仅是期末考试的常客，更是实际开发中数据持久化的核心。

其实这部分知识很好理解，你只需要在脑海里建立一个映射：**把文件流当成 `cin` 和 `cout` 的“亲戚”**。`cin` 和 `cout` 负责和黑乎乎的控制台终端打交道，而文件流则是和硬盘上的文件打交道。

为了使用它们，我们需要引入 `<fstream>` 头文件。这里面有我们需要掌握的“三剑客”：

- **`ofstream` (Output File Stream)**：输出流，专门用来**写**文件（文件版的 `cout`）。
- **`ifstream` (Input File Stream)**：输入流，专门用来**读**文件（文件版的 `cin`）。
- **`fstream`**：可读可写。

按照我们的规矩，我们不看大段代码，而是把文件操作拆解成标准的“四步走”**框架。这不仅是开发规范，更是你在考场上写代码大题的得分模板。我们以**写文件（`ofstream`）为例来拆解：

### 第一步：建立连接（创建并打开）

你想往文件里写东西，就要先实例化一个输出流对象，并告诉它你要打开哪个文件。

```C++
#include <fstream>
// ...

ofstream myFile; 
// 打开文件。如果不写第二个参数，ofstream 默认是以覆盖模式 (ios::out) 打开。
myFile.open("data.txt"); 
```

- **期末避坑点**：如果题目要求“在文件末尾接着写，不能清空原内容”，你必须加上**追加模式**参数：`myFile.open("data.txt", ios::app);`
    

### 第二步：生死攸关的“状态确认”（高频考点）

期末考试的改错题特别喜欢在这里挖坑。文件可能因为权限不够、路径错误或磁盘满了而打开失败。**所以，打开文件后，绝对不能直接暴力读写！**

```C++
// 必须有的防御性代码
if (!myFile.is_open()) {
    cout << "严重错误：文件打开失败！" << endl;
    return -1; // 或者抛出异常
}
```

### 第三步：数据流转（读或写）

确认文件安全打开后，这一步就极其简单了。你怎么用 `cout` 输出，就怎么用你的文件流对象输出。

```C++
// 把文本和换行符“流”进文件里
myFile << "第一行测试数据" << endl;
myFile << 10086 << endl;
```

### 第四步：过河拆桥（关闭资源）

这也是简答题和编程题必看的得分点。操作完毕后，必须断开连接，把内存和系统资源还给操作系统。如果忘了写这一步，数据可能还会滞留在内存缓冲区里，没有真正存进硬盘。

```C++
myFile.close();
```

### 如何正确判断已经读到了文件末尾（EOF）

在处理 `ifstream` 读取时，核心难点在于：**程序怎么知道自己已经读到了文件的最后？**

#### 第一幕：经典的“反面教材”（期末易错题）

很多同学在考场上，或者刚开始写程序时，最直观的逻辑是这样的：既然文件流提供了一个 `eof()` 方法（End Of File），那我就把它当成 `while` 循环的条件。

**错误的逻辑路径：**

1. 只要文件还没到末尾 (`!myFile.eof()`)
    
2. 从文件读取数据
    
3. 处理数据（比如打印出来）
    

**为什么这是个大坑？**

因为 C++ 的 `eof()` 标志**并不是“向前看”的**。当你刚好读完文件最后一行真实数据时，`eof()` 依然是 `false`！只有当你**再次尝试去读**，并且读了个空，系统才会猛然醒悟：“哦，原来到底了”，这个时候 `eof()` 才会变成 `true`。

这就导致了一个致命后果：**循环会多执行一次**。你的程序会把最后一次读取失败时的“残留垃圾数据”（或者最后一行数据重复一次）当作正常数据处理掉。

#### 第二幕：破局之道（标准读取逻辑）

要避开这个坑，我们有两种标准的结构逻辑，对应不同的读取需求。

##### 逻辑一：将其作为条件表达式（“试探性读取”——推荐做法）

这是 C++ 中最优雅、最不容易出错的逻辑。核心思想是：**不要去问文件有没有结束，而是直接去读，读成功了才处理。**

在 C++ 中，读取操作（比如 `getline` 或者 `>>` 运算符）实际上会返回流对象本身。而流对象在 `while` 的条件判断中，会自动转换成布尔值（`true` 或 `false`）。如果读取失败（包括遇到了 EOF），就会返回 `false`。

**结构路径：**

```C++
// 场景 A：按词/按特定类型读取 (比如读入一个接一个的整数)
int number;
while (myFile >> number) { 
    // 逻辑：只要能成功把数据流进 number 里，我就进入循环处理
    // 处理 number 的代码...
}

// 场景 B：按行读取 (最适合处理文本配置文件)
string line;
while (getline(myFile, line)) {
    // 逻辑：只要能成功抽出一整行放到 line 里，就进入循环
    // 处理 line 的代码...
}
```

**这种写法的优势**：完美避开了多读一次的 Bug。因为当读完最后一行真实数据后，下一次 `while` 尝试读取时会立刻失败，循环当场终止，根本没有机会去处理垃圾数据。

##### 逻辑二：前置读取 + 尾部读取（手动控制流）

如果你出于某种复杂的业务需求，非要用 `!myFile.eof()` 作为循环条件，那么你必须采用“先读一次，再判断，循环最后再读一次”的结构。

**结构路径：**

1. 在进入 `while` 循环**之前**，先手动执行一次读取操作。
    
2. 循环条件设为 `while (!myFile.eof())`（或者 `while(myFile)`）。
    
3. 进入循环体，**先处理**刚刚读到的数据。
    
4. 在循环体的**最后一行**，执行下一次读取操作。
    

这样，如果最后一次读取碰壁了，下一轮循环判断 `eof()` 时就会立刻被拦截，不会处理错误数据。不过相比之下，还是**逻辑一**更加简洁干练，也是考卷上最标准无误的答案。
